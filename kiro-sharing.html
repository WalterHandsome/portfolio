Steering 实战

把团队规范写进仓库

Steering 文件放在 .kiro/steering/ 目录下，AI 每次对话自动加载

project.md
# 项目描述 名称: 支付网关服务 技术栈: Python / FastAPI / Vue3 架构: 分层架构 (Router → Service → Repo) # 关键约定 - 所有 API 返回统一 Response 格式 - 数据库操作必须在 Repository 层 - 环境变量通过 Settings 类管理
code-general.md
# 编码规范 ## 核心原则 - 最小必要改动，不加没要求的功能 - 先对齐现有约定，再动手写 - 产出必须是可交付代码 - 单函数 ≤ 60 行 ## 禁止事项 - 不"顺手改进"旁边的代码 - 不臆造不存在的字段和 API
我们项目有 9 个 steering 文件，覆盖 Python / 前端 / 数据库 / 测试 / 重构

Spec 实战

需求 → 设计 → 任务：三阶段规格

以银联消费撤销与退款功能为例

Phase 1
Requirements
用户故事 + 验收标准
→
Phase 2
Design
技术方案 + 接口设计
→
Phase 3
Tasks
可执行的离散任务
requirements.md — 需求 2
## 需求 2：撤销优先、退款降级策略 ### 用户故事 作为支付运营人员，我希望系统自动判断 是否可以撤销，不可撤销时降级为退款。 ### 验收标准 AC-1: WHEN 原交易在当日 THEN 优先发起撤销 AC-2: IF 撤销失败/超时 THEN 自动降级为退款 AC-3: WHEN 原交易跨日 THEN 直接走退款流程 AC-4: IF 退款也失败 THEN 记录异常并通知人工
